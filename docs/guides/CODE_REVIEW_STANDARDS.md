>
> **适用场景**：AI 代码审查、代码自检、代码评审
>
> **最后更新**：2026-04-11
>
> **版本**：v1.0.0
>
> **维护者**：TF2025 开发团队

## 一、项目背景

```
项目类型：Vue3 + TypeScript 前端 / Node.js + Express 后端
数据库：MySQL（云端连接）
认证方式：JWT + 统一权限系统
参考规范：/docs/ 权限系统文档
审查范围：frontend/src/ (前端) 和 backend/src/ (后端)
```

## 二、代码质量

### 1. 冗余代码

- [ ] 未使用的 `import` / 变量 / 函数
- [ ] 重复的逻辑、相似的组件（需提取公共部分）
- [ ] 死代码（永远不会执行的分支）
- [ ] 重复的 TypeScript 类型定义（确保全局唯一）

### 2. 命名规范

- [ ] 变量/函数/组件命名是否语义化
- [ ] 是否遵循 Vue/JS 统一命名约定
- [ ] 常量是否使用大写下划线（`SCREAMING_SNAKE_CASE`）
- [ ] API 命名是否统一（RESTful 规范）

### 3. 类型安全

- [ ] TypeScript 类型定义是否完整
- [ ] 是否存在 `any` 滥用
- [ ] 接口响应类型是否定义
- [ ] Props 类型是否正确传递
- [ ] `as` 断言是否滥用（使用类型守卫更安全）
- [ ] `unknown` vs `any` 使用是否恰当

### 4. 代码风格

- [ ] 注释是否清晰（避免无意义注释）
- [ ] 是否缺少分号或标点符号
- [ ] 缩进是否统一（2空格/4空格）
- [ ] 对象/数组最后一项是否有多余逗号

## 三、潜在 BUG

### 1. 空值处理

- [ ] 未处理的 `null` / `undefined`
- [ ] 可选链 `?.` 和空值合并 `??` 使用是否正确
- [ ] 数组/对象解构是否有默认值

### 2. 边界条件

- [ ] 数组越界访问（`arr[arr.length]`）
- [ ] 除零错误
- [ ] 空数组遍历
- [ ] 表单必填字段校验

### 3. 异步操作

- [ ] Promise 是否有 `catch` 处理
- [ ] `async/await` 错误是否被 `try-catch` 包裹
- [ ] 并发请求是否正确处理
- [ ] 接口超时处理

### 4. Vue 响应式陷阱

- [ ] `ref` / `reactive` 使用是否恰当
- [ ] 响应式数据更新是否触发重渲染
- [ ] `watch` / `computed` 是否有内存泄漏
- [ ] 组件卸载时是否清理定时器/事件监听

### 5. 业务逻辑漏洞

- [ ] 数据闭环是否完整（增删改查链路是否完整）
- [ ] 状态更新后是否刷新列表
- [ ] 删除操作是否有确认提示
- [ ] 表单提交后是否重置状态

## 四、安全漏洞

### 1. 注入攻击

- [ ] SQL 注入（参数化查询是否正确）
- [ ] XSS 攻击（用户输入是否转义）
- [ ] 敏感操作是否有二次验证

### 2. 认证授权

- [ ] JWT Token 是否正确验证
- [ ] 权限校验是否遗漏（`v-permission`）
- [ ] 敏感接口是否鉴权
- [ ] Token 过期处理是否合理

### 3. 数据安全

- [ ] 密码是否安全哈希（bcrypt）
- [ ] 日志是否泄露敏感信息
- [ ] 敏感配置是否使用环境变量

### 4. CORS 配置

- [ ] CORS 配置是否合理
- [ ] 是否暴露不必要的接口

### 1. 数据库查询

- [ ] 大列表是否使用虚拟滚动
- [ ] 是否有不必要的重渲染
- [ ] 重复的 API 请求是否合并

### 3. 资源优化

- [ ] 图片是否压缩/懒加载
- [ ] 防抖/节流是否合理使用
- [ ] 是否有过多的全局状态订阅

### 4. 缓存策略

- [ ] API 响应是否合理缓存

## 八、日志规范

> **重要**：必须使用统一的日志入口，禁止直接使用 console 或底层引擎

### 1. 日志入口检查

| 层级 | 文件 | 业务层是否直接使用 |
|------|------|-------------------|
| 后端统一入口 | `log.js` | ✅ 是 |
| 后端底层引擎 | `logger.js` | ❌ 否 |
| 前端统一入口 | `logger.ts` | ✅ 是 |

### 2. 检查项

- [ ] **后端**：是否使用 `log.js` 而非 `logger.js` 或 `console.log`
- [ ] **前端**：是否使用 `logger.ts` 而非 `console.log`
- [ ] **禁止**：直接使用 `console.log`、`console.info`、`console.debug`
- [ ] **允许**：`log.error()`、`log.warn()`、`log.info()`（重要业务状态）
- [ ] **禁止**：`debugger` 语句
- [ ] **调试日志**：开发完成后应删除或使用 `log.debug()`

### 3. 正确示例

```javascript
// ✅ 后端正确：使用 log.js
const log = require('../utils/log');
log.info('用户登录成功');
log.error('数据库连接失败', error);
log.success('订单创建成功');

// ❌ 后端错误：直接使用 logger.js（底层引擎）
const logger = require('../utils/logger');  // 禁止

// ❌ 后端错误：直接使用 console
console.log('debug');  // 禁止
```

```typescript
// ✅ 前端正确：使用 logger.ts
import { logger } from '@/utils/logger';
logger.info('加载完成');
logger.error('请求失败', error);

// ❌ 前端错误：直接使用 console
console.log('debug');  // 禁止
```

## 十、顺手修复原则 ⭐

> **核心原则**：修改文件时发现其他问题，必须一起修复，不能视而不见

### 1. 必须顺手修复的问题

| 发现的问题 | 处理方式 | 说明 |
|-----------|---------|------|
| 调试日志残留 | 删除 | `console.log`、`log.debug` 等 |
| 内联样式 | 改用 class | 统一样式管理 |
| 重复类型定义 | 删除，使用 `@/types` | 确保类型唯一性 |
| 独立 axios 实例 | 改用 `unifiedApi` | 统一请求管理 |
| 表单验证分散 | 改用 `ValidationRules` | 统一验证规则 |
| 响应式缺失 | 添加移动端适配 | 所有页面必须支持 |

### 2. 修复优先级

1. **安全 > 逻辑 > 性能 > 规范 > 风格**
2. **只修改当前正在处理的文件**，不动其他文件
3. **同类问题需要全局排查**，一并处理

### 3. 禁止忽视

- ❌ "不是本次要改的..."
- ❌ "等其他时候再处理..."
- ❌ "这个看起来没问题..."

**相关文档**：[CLAUDE.md](../../CLAUDE.md) - 开发规范第 12.2.1 节

## 📋 输出格式要求

### 表格格式

| 优先级 | 类别 | 文件位置 | 问题描述 | 建议修复方案 |
|--------|------|----------|----------|--------------|
| 🔴 高 | 安全 | `backend/src/controllers/xxx.js:45` | SQL注入风险 | 使用参数化查询 |
| 🔴 高 | 逻辑 | `frontend/src/views/xxx.vue:120` | 删除后未刷新列表 | 调用 `getList()` |
| 🟡 中 | 性能 | `backend/src/services/xxx.js:88` | N+1查询 | 使用批量查询 |
| 🟡 中 | 规范 | `frontend/src/utils/xxx.ts:10` | any类型滥用 | 定义具体接口类型 |
| 🟢 低 | 风格 | `frontend/src/views/xxx.vue:50` | 命名不规范 | 改为驼峰命名 |
| 🟢 低 | 冗余 | `backend/src/utils/xxx.js:20` | 未使用的导入 | 删除 `console` |

### 输出要求

1. **统计各类问题数量**
   ```
   🔴 高优先级：X 个（安全 X 个、逻辑 X 个）
   🟡 中优先级：X 个（性能 X 个、规范 X 个）
   🟢 低优先级：X 个（风格 X 个、冗余 X 个）
   ```

2. **给出修复优先级建议**
   - 建议按「安全 > 逻辑 > 性能 > 规范 > 风格」的顺序修复

3. **提供可操作的修复代码示例**

4. **标注需要特别注意的业务逻辑问题**
